# Self-Hosted Deployment Plan: VisiGuard on Ubuntu + Self-Hosted Supabase

Goal: Run the full VisiGuard stack (frontend + Supabase backend + WhatsApp bridge + Nginx) on a single Ubuntu 22.04/24.04 server, fully decoupled from Lovable Cloud.

## What changes in the codebase

The app currently hardcodes the Lovable Cloud Supabase project (`bzyvykyuiuihzvhdpxsi`) in three places:

1. `src/integrations/supabase/client.ts` reads `VITE_SUPABASE_URL` / `VITE_SUPABASE_PUBLISHABLE_KEY` from `import.meta.env` тАФ already env-driven, good.
2. `.env` is auto-generated by Lovable and points to the cloud project тАФ we will replace it with a `.env.production` written by the deploy script using your self-hosted values.
3. Edge functions (`supabase/functions/*`) reference `Deno.env.get("SUPABASE_URL")`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, plus Twilio / WhatsApp bridge / Lovable AI secrets тАФ these will be set in the self-hosted Supabase `.env` and via `supabase secrets set`.

Code changes required:

- Add a `.env.example` documenting all required `VITE_*` variables.
- Replace any uses of `LOVABLE_API_KEY` / `lovable-ai-gateway` with a swappable provider:
  - `supabase/functions/anpr-scan/index.ts` (Gemini Vision via Lovable AI). Switch to direct Google Gemini API using `GEMINI_API_KEY` when self-hosted (Lovable AI Gateway is not available off-platform).
- Remove Lovable-specific PWA/iframe guard branches that depend on `lovable.app` host detection тАФ keep them harmless but verify SW registers on your domain.
- Confirm `supabase/config.toml` `verify_jwt` flags are honored by self-hosted CLI (they are).
- Migrations: export current schema from cloud project (via `supabase db dump`) and commit as `supabase/migrations/00000000000000_init.sql` so `supabase db reset` provisions a fresh self-hosted DB identically. Storage buckets (`visitor-photos`, `branding`) recreated via SQL.

## Deployment script (`deploy.sh`) тАФ what it does

A single idempotent bash script you run as root on a fresh Ubuntu box. Prompts once for: domain, supabase API domain, admin email, Postgres password, JWT secret (auto-generated if empty), Twilio creds, Gemini API key, WhatsApp bridge API key.

Steps performed:

1. **System prep** тАФ `apt update`, install `curl git ufw nginx certbot python3-certbot-nginx jq openssl`.
2. **Docker** тАФ install Docker Engine + compose plugin from official repo.
3. **Firewall** тАФ UFW allow 22/80/443; deny everything else.
4. **Clone app repo** to `/opt/visiguard` and check out target branch.
5. **Self-hosted Supabase**:
   - Clone `https://github.com/supabase/supabase` into `/opt/supabase`.
   - Copy `docker/.env.example` тЖТ `docker/.env`, fill in generated `POSTGRES_PASSWORD`, `JWT_SECRET`, `ANON_KEY`, `SERVICE_ROLE_KEY` (generated with `openssl` + `node` JWT script bundled in deploy script), `SITE_URL`, `API_EXTERNAL_URL`, SMTP creds.
   - `docker compose up -d` to bring up the stack.
   - Wait-for-healthy loop on Kong `:8000`.
6. **Apply migrations & seed**:
   - Install Supabase CLI (`curl ... | tar`).
   - `supabase link` against local DB via direct `postgresql://` URL.
   - `supabase db push` to apply `supabase/migrations/*`.
   - Run a `psql` seed creating the `visitor-photos` and `branding` storage buckets and the primary admin user (`bala@sharviinfotech.com`) + HO admin row.
7. **Edge functions**:
   - `supabase functions deploy --no-verify-jwt` for each function (respects `config.toml`).
   - `supabase secrets set` for: `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_WHATSAPP_NUMBER`, `TWILIO_SMS_NUMBER`, `WHATSAPP_BRIDGE_URL`, `WHATSAPP_BRIDGE_API_KEY`, `GEMINI_API_KEY`, `RESEND_API_KEY` (optional).
8. **WhatsApp bridge**:
   - `cd /opt/visiguard/whatsapp-bridge && docker build -t visiguard-wa .`
   - `docker run -d --name wa-bridge --restart=always -p 127.0.0.1:3001:3000 -v /opt/visiguard/wa-data:/data -e API_KEY=... visiguard-wa`
   - First-run QR pairing instructions printed.
9. **Frontend build**:
   - Install Node 20 (`nvm` or NodeSource).
   - Write `/opt/visiguard/.env.production` with `VITE_SUPABASE_URL=https://api.<domain>`, `VITE_SUPABASE_PUBLISHABLE_KEY=<generated anon key>`, `VITE_SUPABASE_PROJECT_ID=self-hosted`.
   - `npm ci && npm run build` тЖТ output to `/var/www/visiguard`.
10. **Nginx**:
    - Site 1: `<domain>` тЖТ static `/var/www/visiguard` with SPA fallback (`try_files $uri /index.html`).
    - Site 2: `api.<domain>` тЖТ `proxy_pass http://127.0.0.1:8000` (Kong) with WebSocket upgrade headers for Realtime.
    - Site 3 (internal-only via Nginx auth or firewall): `wa.<domain>` тЖТ `127.0.0.1:3001` for QR pairing screen.
11. **TLS** тАФ `certbot --nginx -d <domain> -d api.<domain> --agree-tos -m <admin-email> --redirect`.
12. **Systemd units** for: WhatsApp bridge container restart, Supabase compose stack, nightly `pg_dump` to `/opt/backups/`.
13. **Print summary** тАФ URLs, admin login, where logs live, how to re-pair WhatsApp.

## How you run it

```bash

git clone <your-repo-url> /opt/visiguard
cd /opt/visiguard
sudo bash deploy/deploy.sh
# Answer prompts (domain, email, Twilio keys, Gemini key, etc.)
```

For updates after code changes: `sudo bash /opt/visiguard/deploy/update.sh`.

## Out of scope / caveats

- Lovable AI Gateway is not available off-platform. ANPR will use Google Gemini directly тАФ you must supply a `GEMINI_API_KEY`.
- Lovable Cloud secrets manager UI is not available тАФ secrets are managed via `supabase secrets set` and the self-hosted Supabase Studio at `https://api.<domain>/` (protected by Studio basic-auth set in deploy script).
- WhatsApp Web bridge requires keeping a Chromium session alive; QR re-pairing needed if session expires.
- Single-server topology; for HA you would split Postgres, app, and bridge across nodes (not covered).
- Existing data migration from the current Lovable Cloud project to self-hosted is a one-time `pg_dump` / `pg_restore` step documented in `deploy/README.md` but not automated by `deploy.sh`.
