## Goal

Make `deploy.sh` / `redeploy.sh` produce an on-prem VisiGuard that mirrors the cloud project using **only SQL files** (no `pg_dump` binary dump, no cloud network access at deploy time). All uploaded files (visitor photos, branding logos) land on the local Ubuntu filesystem under the project's storage volume.

## Strategy

Two layers of SQL, both checked into the repo:

1. **`supabase/migrations/*.sql`** — schema, RLS, functions, buckets, policies (52 files, already in repo).
2. **`deploy/seed/*.sql`** — data dump as plain `INSERT … ON CONFLICT DO NOTHING` statements, one file per table, generated by a helper script run **once on a machine that can reach the cloud DB**. The output is committed and shipped with the repo, then replayed on the on-prem Postgres.

After redeploy the on-prem DB has identical schema + identical row contents. No `pg_restore`, no `.dump` file, no version-mismatch risk.

Storage files are written by Supabase `storage-api` to `BACKEND_DIR/supabase/docker/volumes/storage/` (a host bind-mount). All four upload sites in the codebase already use the standard `supabase.storage.from(...).upload(...)` API, so they continue to work unchanged — uploads physically land on the Ubuntu disk. A separate seed step copies any existing cloud storage objects into that folder so visitor photos and the branding logo are present after redeploy.

## File-by-file changes

### 1. `deploy/generate-sql-seed.sh` (new, replaces `generate-seed-files.sh` + `export-from-cloud.sh`)

Run **once** on a workstation with access to the cloud DB.

- Inputs (env): `CLOUD_DB_URL`, `CLOUD_SUPABASE_URL`, `CLOUD_SERVICE_ROLE_KEY`.
- For every table in dependency order:
  ```
  locations, departments, gates, screens, role_screen_permissions,
  tenant_settings, email_config, email_templates, vehicle_types,
  employees, profiles, user_location_roles, vehicles, vehicle_entries,
  visitors, accompanying_visitors, visitor_agreements, visitor_watchlist,
  appointments, anpr_events, audit_logs, notifications,
  push_subscriptions, email_logs
  ```
  emits `deploy/seed/30_<order>_<table>.sql` containing:
  ```sql
  INSERT INTO public.<table> (col1, col2, ...) VALUES (...), (...)
  ON CONFLICT (id) DO NOTHING;
  ```
  via `psql --csv` + a small awk/python formatter that handles `jsonb`, `uuid[]`, `timestamptz`, NULLs, and quotes.
- Emits `deploy/seed/20_auth_users.sql` from `auth.users` + `auth.identities` (id, email, encrypted_password, raw_user_meta_data, email_confirmed_at) using the same `INSERT … ON CONFLICT DO NOTHING` shape so existing logins keep working.
- Emits `deploy/seed/10_storage_objects.sql` (rows in `storage.objects`) and downloads the actual object bytes into `deploy/seed/storage/<bucket>/<path>` via the cloud Storage REST API.
- All output is plain text, diffable, and safe to commit.

### 2. `deploy/apply-seed.sh` (new)

Run **inside the on-prem deploy** after migrations are applied.

- For each `deploy/seed/*.sql` (sorted): `docker exec -i supabase-db psql -U postgres -d postgres -v ON_ERROR_STOP=1 -f -`.
- For `deploy/seed/storage/`: `cp -r` into `$BACKEND_DIR/supabase/docker/volumes/storage/stub-XXX/` matching the path layout storage-api expects, then `chown -R 1000:1000`.
- Final `SELECT setval(...)` for any sequences that need bumping (vehicle_id / visitor_id are app-generated, so usually none).
- Idempotent: re-running is a no-op because of `ON CONFLICT DO NOTHING`.

### 3. `deploy/redeploy.sh` (rewrite)

Drop the `--dump` / `--storage` flags. New flow:

```
sudo bash deploy/redeploy.sh                    # schema only (empty DB)
sudo bash deploy/redeploy.sh --with-seed        # schema + committed cloud data
```

Steps:
1. `compose down -v` + wipe `volumes/db`, `volumes/storage`, `wa-bridge` data.
2. Run `deploy.sh` (auto-detects free ports, brings stack up, waits for healthy Postgres).
3. **Apply migrations**: loop `supabase/migrations/*.sql` in filename order through `docker exec supabase-db psql`. Each file wrapped in `BEGIN; … COMMIT;` and skipped if already recorded in a tiny tracking table `public._lovable_migrations(name text primary key, applied_at timestamptz default now())`.
4. If `--with-seed` and `deploy/seed/` exists → `apply-seed.sh`.
5. Run `repair-postgrest.sh` (already exists) to refresh the PostgREST schema cache.
6. Health-check loop: `curl -fsS http://localhost:$API_PORT/rest/v1/locations?select=id&limit=1` until 200.

### 4. `deploy/deploy.sh` (small additions)

- Add a comment block documenting that `volumes/storage/` is the canonical upload location and is bind-mounted into `storage-api`.
- Keep the existing free-port helper (`pick_port`) and `config.env` persistence — unchanged.
- Remove the WhatsApp bridge env's hard-coded reference to cloud Supabase URL; use the on-prem `KONG_HTTP_PORT`.

## Why this satisfies the request

- "Don't dump data from cloud, push it via migration files" → seed `.sql` files are committed alongside `supabase/migrations/`, applied by `psql -f`, no `pg_dump`/`pg_restore` involved.
- "Uploads should work on Ubuntu" → standard Supabase Storage + host bind-mount to `volumes/storage/` already does this; documented and verified in the README and apply-seed step.
- "Symmetric application with all users etc." → `20_auth_users.sql` recreates auth users with original UUIDs and password hashes, `30_…_user_location_roles.sql` re-applies role assignments, and `10_storage_objects.sql` + `seed/storage/` brings the photos.

## Risks and mitigations

- **Password hashes**: `auth.users.encrypted_password` is a bcrypt hash; copying it verbatim works because gotrue uses the same algorithm. Verified by Supabase community recipes.
- **JSONB / array escaping in `INSERT VALUES`**: handled by emitting via Postgres' own `format('%L', col)` from a small `COPY (SELECT format(...)) TO STDOUT` query rather than client-side string building.
- **Sequence drift**: all primary keys are `gen_random_uuid()` defaults — no sequences to reset.
- **Re-running `--with-seed`**: `ON CONFLICT (id) DO NOTHING` makes it safe.
- **Large storage**: if `seed/storage/` grows past Git's comfort zone (say >1 GB), `generate-sql-seed.sh` prints a hint to `tar` it separately and `scp`; the `apply-seed.sh` script picks up either form.
