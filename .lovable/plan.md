

## Fix the "Visitor not found" error when scanning the check-in QR

### Root cause
The check-in QR generated by `approve-visitor` encodes:
```json
{ "visitorId": "<UUID>", "name": "...", "action": "checkin" }
```
where `visitorId` is the visitor's **UUID** (`visitors.id`, e.g. `c2298e95-d536-4126-9166-...`).

But `CheckInOut.handleQrScan` looks it up like this:
```ts
.from('visitors')
.select(...)
.eq('visitor_id', data.visitorId)   // ← wrong column
.single();
```
`visitor_id` is the **human-readable code** (e.g. `VIS-0C1BC226-A734`), not the UUID. The query returns no rows → toast says "Visitor not found".

A second, smaller bug: even if the lookup matched, the handler only opens the photo dialog when `status === 'scheduled'`. After a fresh approval the status is indeed `'scheduled'`, so this part is fine — but for self-service/host-pending cases the status is `'pending_approval'`, which currently falls into the silent "already checked out" branch. We should treat `pending_approval` as "needs host approval first" with a clear message instead.

### Fix — `src/pages/CheckInOut.tsx` only

1. In `handleQrScan`, accept **either** column. Try the UUID first (since that's what the approval QR uses), then fall back to the human-readable `visitor_id` (used by older badge QR codes printed at the gate):
   ```ts
   const { data: byUuid } = await supabase
     .from('visitors')
     .select(`*, host:employees(*, department:departments(*)), department:departments(*)`)
     .eq('id', data.visitorId)
     .maybeSingle();

   const visitorData = byUuid ?? (await supabase
     .from('visitors')
     .select(`*, host:employees(*, department:departments(*)), department:departments(*)`)
     .eq('visitor_id', data.visitorId)
     .maybeSingle()).data;
   ```
   Use `maybeSingle()` so a no-row result no longer throws — only a real lookup miss shows "Visitor not found".

2. Default `data.action` handling — when **no** `action` is present **and** the visitor is `'scheduled'`, treat it as a check-in (current behavior). When `action === 'checkin'` is explicit (the new approval QR), do the same: open the camera dialog.

3. Add a friendly `pending_approval` branch:
   ```ts
   } else if (visitor.status === 'pending_approval') {
     toast.warning('This visitor is still awaiting host approval.');
   ```

4. No changes to `QrScanner`, no changes to `approve-visitor`, no DB changes. The same fix automatically lets `VehicleGate` continue to work because it has its own scan handler (different file, different lookup keyed by vehicle).

### Files touched
- **Edit** `src/pages/CheckInOut.tsx` only — make the visitor lookup accept either `id` (UUID, used by the new approval QR) or `visitor_id` (human code, used by printed badges), use `maybeSingle()`, and add a clear message for `pending_approval`.

### What you'll do after the fix

```text
1. Approve a freshly created visitor (host taps Approve in WhatsApp).
2. Open Check-In/Out → Scan tab → Start Scanning.
3. Scan the visitor's check-in QR (the one received on WhatsApp/email).
   → Toast: "QR scanned: <name>"
   → Camera dialog opens for photo capture
   → After capture: status becomes checked_in, badge prints,
     visitor receives the CHECKOUT QR via WhatsApp + email + SMS.
4. Scan a printed badge QR (legacy format with VIS-XXXX-YYYY) from another
   visitor → still works (fallback lookup by visitor_id).
5. Scan a QR for a still-pending visitor → friendly toast:
   "This visitor is still awaiting host approval."
```

### Out of scope
- VehicleGate scanner (separate file, different model).
- Self-service portal (uses the same `approve-visitor` → benefits automatically).
- QR scanner camera/library code.
