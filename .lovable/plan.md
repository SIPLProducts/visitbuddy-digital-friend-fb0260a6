## Root cause (found)

`deploy/seed/40_visitors.sql` line 355 ends mid-string:
```
... 'Kewaunee', 'Official
```
no closing `'`, no `);`. Line 356 starts a new `INSERT INTO public.visitors ... VALUES ('1b5eb6aa-...`. Postgres treats line 355 + 356 as one statement: the unterminated `'Official\nINSERT INTO ... VALUES('` string ends at the next `'`, then the parser sees `1b5eb6aa-...` at the top level and reports `trailing junk after numeric literal at or near "1b"`.

This happened because `deploy/generate-seed-files.sh` does:
```bash
pg_dump --data-only --inserts --column-inserts ... | grep -E '^(INSERT|SELECT ...)'
```
When any text value (e.g. `purpose`) contains a newline, `pg_dump` legitimately wraps the value across multiple lines — but `grep '^INSERT'` keeps only the first line and drops the rest, leaving an unterminated string literal.

The same bug also affects any other seed file whose source rows had newlines in text fields (likely `appointments.notes`, `visitor_watchlist.reason`, `audit_logs.details`, etc.).

## Plan

1. **Add `deploy/sanitize-seed.sh`**
   New script that scans every `deploy/seed/*.sql` and rejoins continuation lines so each statement ends with `;`. It treats only lines starting with `INSERT `, `SELECT `, `BEGIN`, `COMMIT`, `TRUNCATE`, `--`, or blank as new statements; everything else is appended (with a `\n`) to the previous statement. This rebuilds the dropped lines from `pg_dump` output that `grep` chopped off.

   Usage:
   ```bash
   sudo bash deploy/sanitize-seed.sh
   sudo bash deploy/import-seed.sh
   ```

2. **Fix `deploy/generate-seed-files.sh`**
   Replace the lossy `grep '^(INSERT|SELECT...)'` filter with a multi-line aware filter (small Python/awk inline) that only strips `pg_dump`'s comment/SET preamble but preserves complete statements regardless of internal newlines. This prevents the bug from re-occurring next time seeds are regenerated.

3. **Hook sanitizer into `deploy/import-seed.sh`**
   Before the import loop, run the sanitizer in-place. That way users on already-broken servers don't need to remember an extra step — `import-seed.sh` self-heals.

4. **Update `deploy/README-troubleshooting.md`**
   Add section for `trailing junk after numeric literal` → run sanitizer or re-pull. Mention this only happens for seeds generated by an older `generate-seed-files.sh`.

## Technical details

The sanitizer logic (Python):
- Read file as text.
- Walk lines; if the line starts with one of the statement keywords (or is blank/`--`), flush the current buffer and start new.
- Otherwise append `\n` + line to current buffer.
- Write back joined output.

This is safe and idempotent — already-correct rows have no continuation lines, so they pass through unchanged.

For the `generate-seed-files.sh` fix, the simplest correct filter is:
```bash
pg_dump ... | python3 -c '
import sys
buf=[]
for line in sys.stdin:
    if line.startswith(("--","SET ","SELECT pg_catalog.set_config")) or line.strip()=="":
        continue
    buf.append(line)
sys.stdout.write("".join(buf))
'
```
which keeps full INSERTs even when they span lines.
